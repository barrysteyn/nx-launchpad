Generate a new Python application in this Nx monorepo using the workspace generator.

Follow these steps:

1. **Ask the user for the following before running anything:**
   - App name (must be kebab-case, e.g. `my-python-app`)
   - Short description of the app
   - Python version (default: `3.12.3`)
   - Include AWS infrastructure? (yes/no, default: yes)
   - If yes: infrastructure type (`lambda` or `ecs`, default: `lambda`)
   - If lambda: include API Gateway? (yes/no, default: yes)

2. **Run the generator** with the collected answers:
   ```
   npx nx generate @nx-launchpad/tools:python-app <app-name>
   ```

3. **Verify the generated files:**
   - `apps/<app-name>/src/<app_module>/main.py` — has both `handler(event, context)` and `main()`
   - `apps/<app-name>/pyproject.toml` — name, description, and script entry point are correct
   - `apps/<app-name>/uv.lock` — exists (created by `uv sync` during generation)
   - `apps/<app-name>/infra/` — present if infrastructure was selected

4. **Run the generated project's targets to confirm the scaffold works:**
   ```
   npx nx run <app-name>:lint
   npx nx run <app-name>:format
   npx nx run <app-name>:test
   npx nx run <app-name>:build
   ```
   If infra was generated, also run:
   ```
   npx nx run <app-name>:tf-validate
   ```

5. **Verify and fix the lock file** — run:
   ```
   npm ci --legacy-peer-deps
   ```
   If this fails with a lock file sync error, regenerate it:
   ```
   npm install --legacy-peer-deps
   ```
   Then re-run `npm ci --legacy-peer-deps` to confirm it passes.

6. **Commit the result** using two commits:
   - One for the generated app files
   - One for `package-lock.json` if it changed

**Conventions to enforce:**
- App name must be kebab-case
- Python module name is the app name with hyphens replaced by underscores
- Always commit `pyproject.toml` and `uv.lock`
- Never commit `requirements.txt` — it is gitignored and generated by the `build` target
- Never commit `.venv/` or `__pycache__/`
